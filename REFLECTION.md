# 專案反思與心得

## Gemini 對話機器人 — LangChain + Streamlit

---

## 一、專案概述

本專案從零開始打造了一個基於 **Google Gemini 2.5 Flash** 的多模態對話機器人，透過 **LangChain** 框架串接大型語言模型，並以 **Streamlit** 建構 Web 互動介面。從最初的純文字 CLI 對話，逐步擴展至支援圖片、PDF、文件分析，最終加入動態角色切換與本地檔案勾選機制。

---

## 二、開發歷程回顧

### 階段一：基礎建構（v1.0）

一開始的目標很單純——讓程式能透過 Gemini API 進行多輪對話。使用 LangChain 的 `RunnableWithMessageHistory` 快速實現了對話記憶功能，整個核心邏輯不到 100 行。

**學到的事：**
- LangChain 的抽象層讓串接 LLM 變得非常簡潔，不需要直接處理 API 細節。
- `dotenv` 管理 API 金鑰是基本但重要的安全實踐。

### 階段二：多模態擴展（v1.1）

加入圖片、PDF、文件傳送功能是第一個重大挑戰。原本使用 `RunnableWithMessageHistory` 的架構無法直接支援多模態的 `HumanMessage`，因此重構為手動管理訊息歷史的方式。

**學到的事：**
- Gemini 的多模態能力強大，支援原生 PDF 理解，不需要額外的文字提取工具。
- 架構設計需要考慮未來擴展性——一開始選用的便捷封裝（`RunnableWithMessageHistory`）在需求變複雜時反而成為限制。
- Base64 編碼是傳送二進位檔案給 API 的標準做法，但要注意檔案大小對記憶體的影響。

### 階段三：Web 介面化（v1.2）

從 CLI 轉向 Streamlit Web 介面，大幅提升了使用體驗。拖放上傳、即時圖片預覽、聊天氣泡等功能讓互動更直覺。

**學到的事：**
- Streamlit 的 `st.chat_message` 和 `st.chat_input` 元件讓聊天介面的開發效率極高。
- `st.session_state` 是 Streamlit 中管理狀態的關鍵，需要同時維護「顯示用」和「LangChain 用」兩套歷史。
- `@st.cache_resource` 避免每次頁面重新渲染時重複初始化 LLM，是效能優化的重要技巧。

### 階段四：角色切換與檔案勾選（v1.3）

最後加入的動態 System Prompt 切換和本地檔案勾選機制，讓機器人從單一用途工具進化為多用途平台。

**學到的事：**
- System Prompt 的設計對 AI 回覆品質影響巨大——不同角色的 prompt 需要針對性地調整指令和語氣。
- 本地檔案掃描功能需要考慮安全性和效能（過濾不支援的格式、限制檔案大小）。
- 使用者體驗的細節很重要：全選/取消全選按鈕、檔案大小顯示、類型 emoji 等小功能都能提升易用性。

## 三、技術反思

### 做得好的部分

1. **漸進式開發**：從 CLI → 多模態 → Web UI → 進階功能，每一步都建立在前一步的基礎上，降低了複雜度。
2. **關注點分離**：檔案處理邏輯（`build_file_content`）與 UI 邏輯分開，方便維護和測試。
3. **使用者體驗**：即時預覽、檔案類型 emoji、錯誤提示等細節讓介面更友善。
4. **工作紀錄**：透過 `CHANGELOG.md` 持續記錄每次變更，保持開發軌跡清晰。

### 可改進的部分

1. **缺乏測試**：目前沒有單元測試或整合測試，日後功能越多越容易出現迴歸問題。
2. **對話歷史無持久化**：目前使用 `InMemoryChatMessageHistory`，關閉程式後對話紀錄就消失。可以考慮接入 SQLite 或檔案儲存。
3. **錯誤處理可更細緻**：目前大多數錯誤都是統一的 `except Exception`，可以針對 API 限額、網路逾時等情境提供更具體的錯誤訊息。
4. **Streaming 回覆**：目前是等待完整回覆後才顯示，加入 streaming（逐字顯示）可以大幅提升回應感受。
5. **安全性**：API 金鑰直接存在 `.env` 檔案中，正式部署時應改用環境變數或密鑰管理服務。

## 四、Vibe Coding 開發模式心得與提示詞優化

本專案的開發過程大量採用了 **Vibe Coding** 的模式——以自然語言向 AI 描述需求，由 AI 產出程式碼，再進行審閱、微調與迭代。這種「對話式開發」徹底改變了傳統的編碼節奏：不再是先構思每一行語法再敲鍵盤，而是先釐清「我要什麼」，用一段清楚的描述讓 AI 快速生成骨架，接著在實際運行後根據結果持續修正。整個過程更接近「設計→驗證→微調」的循環，而非「撰寫→除錯」的線性流程。

在這之中，**提示詞（Prompt）的品質直接決定了產出的品質**，這是 Vibe Coding 最核心的學習。一開始我往往只給出模糊的功能描述，例如「幫我加上檔案上傳功能」，結果 AI 雖然會產出可運作的程式碼，卻可能遺漏邊界條件或與現有架構不一致。後來我學會在提示中加入更多上下文——像是目前的技術棧、已有的函式介面、預期的使用者操作流程——產出的程式碼品質立刻有明顯提升。我也發現，**將大需求拆成小步驟逐一交付，比一次性描述整個功能更容易得到準確的結果**，這與軟體工程中「小步快跑」的敏捷精神不謀而合。

另一個重要的發現是，Vibe Coding 並不代表開發者可以完全放手。AI 產出的程式碼仍然需要人類的判斷力來審閱架構合理性、確認安全性考量、以及決定哪些設計取捨符合專案的長期方向。提示詞優化的過程本質上就是一種「溝通能力的鍛鍊」——如何用最精確的語言表達需求，讓 AI 成為高效的開發夥伴，而不只是程式碼生成器。這份能力在未來 AI 輔助開發越來越普及的趨勢下，將會是每位開發者不可或缺的核心素養。

## 五、未來展望

- **Streaming 輸出**：使用 LangChain 的 streaming 功能，讓回覆逐字呈現。
- **對話歷史持久化**：儲存到本地資料庫，支援查看歷史對話。
- **多 Session 管理**：支援建立多個對話分頁，每個分頁有獨立的上下文。
- **RAG（檢索增強生成）**：結合向量資料庫，讓 AI 能參考自己的知識庫回答問題。
- **部署上線**：使用 Streamlit Cloud 或 Docker 部署，讓其他人也能使用。

---

## 五、總結

這個專案讓我深刻體會到，現代 AI 開發的門檻已經大幅降低。透過 LangChain 和 Streamlit 這樣的工具，即使是相對簡單的程式碼也能打造出功能豐富的 AI 應用。然而，真正的挑戰不在於「讓它動起來」，而在於如何設計好的使用者體驗、如何讓架構保持可擴展、以及如何在快速迭代中保持程式碼品質。

從 v1.0 到 v1.3 的演進過程，每一次功能擴展都帶來了新的設計決策和取捨，這正是軟體開發最有趣的地方。
